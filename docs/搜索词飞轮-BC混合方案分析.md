# 搜索词飞轮 — B+C 混合方案分析

## 一、方案 B+C 流程回顾

```
一轮飞轮产生 50 个候选词

  阶段B：AI分析父级搜索结果中已有的信号（0次额外搜索）
    │   从"自行车"的搜索结果里，看候选词"瓜车"出现了几次、
    │   这些出现的商品有没有soldCount、want多少、价格多少
    │
    ├─ 信号足够判断 → 直接评分 → pass/watch/discard
    │
    └─ 信号不足判断 → 标记为"pending"，进入阶段C

  阶段C：对pending词做1页轻量搜索（每个1次搜索，加间隔）
    │   numFound + soldCount + 卖家结构 → 精准评分
    │
    └─ pass入队 / watch保留 / discard淘汰
```

---

## 二、好处

### 1. 数据驱动，不凭感觉

```
常识判断（旧方案）：
  "瓜车好像挺火的，给7分吧" ← 脑补

数据分析（BC方案）：
  "自行车搜索结果20条中瓜车出现3次，其中2条有soldCount（23、65），
   1条'已售100+'。在售47件中有28件有售出记录=60%售出率。
   数据说它真实、需求强、竞争低 → 8.2分" ← 证据链
```

这个区别是根本性的——前者是猜，后者是验证。

### 2. 风控成本可控

- 阶段B：0次搜索，AI处理50个候选词，token成本~¥0.1
- 阶段C：只对阶段B筛剩的20个搜1页，加15秒随机间隔，总共5分钟，风控风险低
- 对比全量搜索50次：15分钟连续调用，必然触发风控

### 3. 误判率可控

阶段B的误判方向是确定的：
- 可能漏判：候选词在父级搜索中出现的商品恰好没售出记录，但搜索这个词本身会发现大量需求 → 进入阶段C后纠正
- 不会误判高分：阶段B给的评分是保守的，信号不足就降级到pending

### 4. 每轮产出都是经过验证的

入队的每个词都有至少一次搜索验证（要么从父级搜索结果中验证，要么经过阶段C独立搜索验证）。没有光凭常识进词库的。

---

## 三、坏处

### 1. 阶段B对父级搜索的依赖

```
场景：父级搜索"自行车"20条结果中，没有一条提到"死飞倒刹"

阶段B：AI看不到任何关于"死飞倒刹"的信号 → 只能给低分
阶段C：如果它没进pending，就永远不会被搜一次验证

风险：品类强相关的候选词不会被漏（因为会在父级搜索中出现），
      但跨品类的候选词可能被漏。

解法：在阶段B中，如果AI认为某个词"品类不同但可能高价值的"，
      即使信号不足也强制进入阶段C。
      判断标准：词的结构有品牌+型号+参数特征 → 值得验证
```

### 2. 阶段B的信号可能不完整

```
父级搜索"自行车"返回的20条里只有3条提到"瓜车"。

这3条的数据：
  - 瓜车1: soldCount=0, want=12    
  - 瓜车2: soldCount=23, want=5    ← 有售出
  - 瓜车3: soldCount=0, want=8

阶段B的AI基于这3条判断：
  "售出率33%，一般" ← 但3条样本太小

阶段C独立搜索"瓜车"返回20条：
  售出率60%，numFound=47 → 阶段B低估了

风险：父级搜索结果中的小样本可能导致阶段B的评分偏保守

好处：保守偏低的评分 = 漏判 → 进入阶段C改正，不会产生产品影响
      如果评分偏高，进了词库但实际不行，浪费的是采集资源
      → 所以宁可保守偏低
```

### 3. 两阶段增加一次 AI 调用

```
旧方案：一次AI调用，常识评分 → 直接出结果
新方案：阶段B一次AI调用（信号提取+评分），阶段C一次AI调用（搜索数据分析）

但阶段B的AI调用比旧方案便宜（分析已有数据 vs 凭空推理，token更少）
阶段C的AI调用只处理pending词（约20个 vs 50个）
总成本变化不大：~¥0.2-0.3/轮 vs ~¥0.1-0.15/轮
```

### 4. 阶段C需要等待

```
阶段B：即时，AI调用1-2秒
阶段C：需要搜索+等待间隔，20个词 × 15秒 = 5分钟

坏处：一轮飞轮从"几秒出结果"变成"几分钟出结果"
缓解：这5分钟里用户可以做别的事，飞轮在后台跑
```

## 四、六个控制机制重设计

原来的六项在数据驱动的BC方案下需要调整：

### 机制一：品类多样性 — 数据验证版

```python
# 旧：AI凭常识判断品类
# 新：从已有搜索数据中提取品类分布

def extract_category_distribution(search_data: dict) -> dict:
    """
    从搜索结果的标题和cpvLabels中提取品类分布。
    
    自行车搜索20条结果：
    → 父级搜索标题中的品类信号分布
    → 直接从数据里算，不靠AI猜
    """
    categories = Counter()
    for item in search_data.get("search_items", []):
        title = item.get("title", "")
        # 从标题中提取品类词
        # "捷安特ATX830山地车" → "山地车"
        # "瓜车gravel砾石公路车" → "瓜车"
        cat = classify_from_title(title)
        categories[cat] += 1
    return categories
```

品类多样性保护触发条件不变：单品类 > 40% → 自动提权，但现在基于真实数据。

### 机制二：通过率异常 — 现在有意义了

之前你问"什么叫异常通过率"。在数据驱动的BC方案里：

```
正常情况：
  一轮50个候选词 → 阶段B筛剩20个 → 阶段C过后8个pass
  通过率 = 8/50 = 16%

异常偏高（通过率 > 50%）：
  可能原因：父级搜索结果恰好集中在高价值品类，或阶段B阈值设得太低
  
  AI诊断（不是人工判断）：
  "本轮25个候选来自'自行车'品类，其中15个是品牌+型号格式，
   在父级搜索中均有soldCnt信号。品类内词质量高但品类单一。
   建议：下轮增加其他品类的种子词。"

异常偏低（通过率 < 3%）：
  可能原因：父级搜索结果本身质量差（搜索词太宽泛），
  或候选词提取出现了大量噪声
  
  AI诊断：
  "本轮候选词中73%在父级搜索结果中无任何信号，可能原因：
   1) 父级搜索词'自行车'返回结果偏向低端车，高价细分未出现
   2) 候选词提取混入了大量标题修饰语而非品类词
   建议：用更精准的父级词做下一轮搜索"
```

异常通过率现在有用——因为它反映的是候选词质量和父级搜索质量的真实问题。

### 机制三：队列硬上限

不变。待采集队列 > 80 个时，新词降级为 watch。

### 机制四：边际价值衰减

```python
# 旧：基于评分均值的衰减
# 新：基于词库覆盖率的衰减

def check_diminishing_returns(word_registry: dict, new_candidates: list) -> dict:
    """
    如果新候选词中 > 60% 是已有词库的同义词/近义词，
    说明品类覆盖已接近饱和，飞轮边际价值在衰减。
    
    这个判断基于实际的数据比对（字符串相似度、搜索结果重叠度），
    不需要AI猜。
    """
    similar_count = 0
    for candidate in new_candidates:
        # 跟已有词库比对
        for existing_word in word_registry:
            if word_similarity(candidate, existing_word) > 0.8:
                similar_count += 1
                break
    
    overlap_rate = similar_count / len(new_candidates)
    
    if overlap_rate > 0.6:
        return {
            "warning": f"新候选词中 {overlap_rate:.0%} 与现有词库高度重复",
            "suggestion": "当前品类可能已饱和，建议人工注入全新品类的种子词",
            "auto_action": "暂停飞轮，通知用户"
        }
    
    return {"healthy": True}
```

### 机制五：自动降级 — 基于产出追踪

```python
def auto_degrade(word_registry: dict):
    """
    不是基于评分趋势，而是基于实际产出：
    
    一个词进入词库后，如果走完完整流水线（行情→选品→货源）后：
    - 30天未产出S/A级商品 → expired
    - 连续3次产出C/D级商品 → 降级为watch
    
    判断依据是硬数据（有产出/没产出），不依赖AI评分。
    """
    for word, info in word_registry.items():
        if info["state"] == "active":
            products = info.get("produced_products", [])
            
            if len(products) == 0 and info["days_since_added"] > 30:
                info["state"] = "expired"
                
            elif len(products) >= 3:
                recent_3 = products[-3:]
                if all(p["grade"] in ("C", "D") for p in recent_3):
                    info["state"] = "degraded"
                    info["degrade_reason"] = "连续3次产出C/D级商品"
```

### 机制六：飞轮自动暂停/重启

```python
PAUSE_CONDITIONS = {
    "saturation":      "连续2轮边际贡献衰减 > 60%",
    "pipeline_full":   "待采集队列 > 80",
    "quality_drop":    "本轮pass的词中 > 30% 在下一轮产生了0个有效商品",
    "user_override":   "手动暂停",
}

def auto_restart_check(state):
    """暂停后的自动重启条件"""
    if state.auto_paused:
        # 队列消化到 < 30 时自动重启
        if state.queue_size < 30:
            return True
        # 或者用户注入了新品类种子词
        if state.has_new_seed_words:
            return True
    return False
```

## 五、飞轮和采集是否可以同时进行

### 答案是：可以，但不要共享同一个 Frida 会话

飞轮和采集是两个不同的任务：

| | 飞轮（阶段C） | 采集流水线 |
|---|---|---|
| 干什么 | 对候选词做轻量搜索验证 | 行情采集 + 选品采集 + 货源查找 |
| 调用 | 搜1页，不进详情 | 搜索+行情+详情+评论 |
| 频次 | 20次搜索/轮，每轮间隔可长 | 按用户操作 |
| 资源 | 轻，1次RPC/词 | 重，每词4-8个RPC |

### 三种并发模式

**模式一：串行（保守，推荐起步用）**

```
飞轮阶段C ──完成──→ 采集流水线 ──完成──→ 飞轮下一轮
```

最简单，不会互相干扰。适合早期跑通流程。

**模式二：分时复用（实用）**

```
飞轮阶段B（AI处理，不需要设备）可以在任何时候跑
飞轮阶段C（轻量搜索）可以用采集流水线的自然间隔来跑

采集流水线在做"行情采集"时：
  每个词之间有10-15秒的关键词间隔 ← 这个间隙
  飞轮阶段C利用这些间隙插入轻量搜索

伪代码：
  for kw in collection_pipeline.keywords:
      collect_market(kw)  # 主要工作
      
      # 自然间隔：飞轮利用这个间隙
      if flywheel.has_pending_searches():
          flywheel.do_one_search()  # 插入一次轻量搜索
      
      sleep(keyword_interval)  # 原本的间隔
```

**模式三：完全并行（最优但需要设备引擎支持）**

```
两个独立的采集队列，用不同的设备信息：

设备引擎维护两个虚拟会话：
  - 主会话：采集流水线（行情+选品）
  - 副会话：飞轮阶段C（仅轻量搜索，优先级低）

副会话只在主会话空闲时干活，而且遵守更激进的风控限制：
  - 主会话最小间隔：5秒
  - 副会话最小间隔：20秒
  - 主会话工作时副会话暂停
  - 副会话单次最多执行5次搜索，然后等主会话空闲 > 30秒才能继续
```

### 推荐方案

起步用**模式一**（串行），原因：
- 简单，不需要改动 device_engine
- 飞轮不是高频操作（一轮飞轮管很久）
- 用户控制感强（看到飞轮跑完再看采集结果）

跑通后可以考虑**模式二**（分时复用），改动量小：
- 在 collection_engine 的关键词间隔回调里加飞轮的搜索队列
- 不需要改动 Frida bridge，只是调度逻辑

## 六、一句话总结

BC方案的核心价值：**AI不凭常识猜，而是分析手上已有的数据。不够判断的时候才针对性地搜一次验证。** 搜索次数从全量50次降到阶段C的20次，风控可控。控制机制从基于评分趋势改到基于实际数据验证。飞轮和采集可以分时复用同一个Frida会话，起步阶段先串行。
